## 2. Findings / fixes during B4

1. **`license` SPDX vs `License ::` classifier (PEP 639).** scikit-build-core refused the
   metadata because `pyproject.toml` had both `license = "LGPL-2.1-only"` and a
   `License :: OSI Approved …` classifier. Removed the classifier (the SPDX field is
   authoritative).

2. **`_build_info.py` was missing from the wheel.** It is generated by CMake and git-ignored,
   so scikit-build-core's `packages` collection skipped it while still shipping the
   `_build_info.py.in` template — `import pysmesh` then failed at `from . import _build_info`.
   Fixed with `install(FILES ${PKG_DIR}/_build_info.py DESTINATION pysmesh)` (CMake install is
   not filtered by `.gitignore`). Verified the installed wheel now imports from a clean dir.

3. **delvewheel is the DLL-bundling step**, run in CI after the raw wheel: it vendors
   `_core.pyd`'s dependency DLLs into `pysmesh.libs` (name-mangled) and is told to
   `--exclude vtk*.dll`. Result: OCCT + Boost bundled, VTK left to the host — the architecture
   guarantee, now machine-checked by `ci/check_wheel.py` (44 DLLs, zero `vtk*`).

4. **The wheel/dev-layout reconciliation** (deferred since B2) is resolved: `conftest.py`
   imports `pysmesh` if installed and only falls back to `src/` for a dev build, so the same
   suite validates the installed wheel (CI) and the in-tree build (local).

5. **`GIT_SHA` wired** — `_build_info.GIT_SHA` is now the short commit (e.g. `efc9fda04`),
   `"unknown"` outside a checkout.

## 4. Known gaps / notes

- **GitHub Release publication** is a tag/maintainer action; the CI job
  (`softprops/action-gh-release` on `refs/tags/v*`) attaches the repaired wheel automatically.
- **`flux-vtk-version.txt`** is consumed by `ci/check_vtk_pin.py` if present; publishing it
  (or reading flux's `environment.yml`) is a flux-side decision — until then the check
  validates env-pin vs `_build_info` self-consistency.
- **abi3 tag** (`cp313-abi3`) comes from `py-api = "cp313"`; the wheel targets CPython 3.13
  (flux's runtime) and is not portability-tested below 3.13.
- The **CI has not been run on GitHub** (no push here); the workflow is written against the
  exact local sequence that passed, but first-run tweaks (action versions, micromamba cache)
  may be needed.
