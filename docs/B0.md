## 1. Header audit — corrections and confirmations to `plan.md`

Full detail in `docs/upstream_notes/{ViscousLayers,ProxyMesh,SMESHDS_Mesh}_notes.md`.
Summary of everything that changes downstream work:

### 1.1 `StdMeshers_ViscousLayers` (confirmed + 1 naming correction)

- **Naming correction:** the boundary-shape setter is `SetBndShapes(const std::vector<int>& shapeIds, bool toIgnore)`, **not** `SetFaces(ids, isToIgnore)` as written in `plan.md` B3 task 1. Use `SetBndShapes` in `bindings/viscous.cpp`.
- `ExtrusionMethod` has exactly 3 values, in this fixed order (persisted, don't reorder): `SURF_OFFSET_SMOOTH`, `FACE_OFFSET`, `NODE_OFFSET`.
- `Compute(SMESH_Mesh&, const TopoDS_Shape&, bool toMakeN2NMap=false) const → SMESH_ProxyMesh::Ptr`. Confirmed signature matches `plan.md` modulo the extra optional 3rd param (leave default `false` for Tier-1).
- **Requires a SOLID.** `Compute()` explores `theShape` for `TopAbs_SOLID` and fails immediately if none exists. `compute_viscous_layers` must validate the input `Shape` has ≥1 solid and raise a clear `PysmeshError` before calling into SMESH, rather than surfacing SMESH's generic error.
- **Group creation is automatic.** `Compute()` internally calls the static `CreateGroup()` itself (using `SMDSAbs_Volume`) whenever `SetGroupName()` was given a non-empty name. The binding does not call `CreateGroup` directly — set the name, call `Compute()`, then look the group up via `mesh.GetGroups()`. `plan.md`'s B3 task 3 already assumed this correctly.
- **Design decision, not just a note: require non-empty `group_name` in `VLParams`.** Without a group name no group is created and prism harvest would need a full-mesh volume-element scan instead — avoid that fallback path entirely by making `group_name` mandatory at the Python API level.

### 1.2 Compute-error surfacing — genuine gap in `plan.md`, resolved

**This is the most important finding of B0.** `plan.md` B3 task 2 says: "On failure,
collect `SMESH_ComputeError` messages and surface them as `PysmeshError.details`" — as
if the returned `SMESH_ProxyMesh::Ptr` (or some field near it) carries the error. It does
not.

Traced through `StdMeshers_ViscousLayers::Compute()` → `_ViscousBuilder::Compute()` →
`_ViscousBuilder::error()`: on hard failure, the public `Compute()` wrapper **discards**
the `SMESH_ComputeErrorPtr` it receives internally and returns an empty `Ptr()`. The only
place the actual error text (`_error->myComment`, prefixed `"Viscous layers builder: "`)
is stored is on the **per-solid submesh's** compute-error slot:
`mesh.GetSubMesh(solidShape)->GetComputeError()`.

**Binding requirement for B3:** after `Compute()` returns, always walk
`TopExp_Explorer(shape, TopAbs_SOLID)` and call `GetComputeError()` on each solid's
submesh, regardless of whether `Compute()` returned null or non-null — null return means
hard failure (collect all non-OK messages, raise `PysmeshError`), non-null with some
submesh error means partial failure (route those solids' faces into
`failed_face_ids` instead of raising). Full detail with line numbers in
`docs/upstream_notes/ViscousLayers_notes.md` §"`Compute()` behavior".

### 1.3 Shape-index hazard — genuine gap in `plan.md`, resolved

**Second most important finding.** `plan.md` B2 tasks 3-4 phrase `classify_on_face` and
`add_triangles` as calling `SetNodeOnFace(node, face_id, u, v)` /
`SetMeshElementOnShape(elem, face_id)`, implying the binding's own `face_id` (a 1-based
ordinal from a `TopAbs_FACE`-only explorer, per `Shape.faces()`) can be passed to
SMESHDS's `int Index` overloads of these methods.

**It cannot, safely.** `SMESHDS_Mesh::ShapeToMesh()` populates its internal index map via
`TopExp::MapShapes(shape, map)` — a whole-shape, all-kinds traversal (solids before
shells before faces before edges before vertices) that has no fixed relationship to a
faces-only ordinal. Passing our `face_id` into the `int Index` overload would silently
classify nodes onto the wrong sub-shape for any shape containing solids/shells (i.e.
essentially every shape this project cares about, since VL requires solids per §1.1)
This is a silent-corruption class bug, not a crash: it passes `Mesh.validate()` (which
only checks *that* a node is classified, not *onto what*) and only surfaces as physically
wrong output.

**Binding requirement for B2:** `Mesh` must retain the same explorer-ordered
`TopoDS_Face`/`TopoDS_Edge`/`TopoDS_Vertex` lists that `Shape.faces()/.edges()/.vertices()`
built (either by holding a reference to the originating `Shape` or copying its ordered
vectors), resolve `face_id → TopoDS_Face` internally, and always call the
**shape-reference overloads** (`SetNodeOnFace(node, const TopoDS_Face&, u, v)`,
`SetMeshElementOnShape(elem, const TopoDS_Shape&)`) — never the `int Index` overloads.
Confirmed as the idiom every real caller in the codebase uses (see
`docs/upstream_notes/ProxyMesh_notes.md`'s NETGENPlugin cross-reference: real code always
re-derives `TopoDS_Face&` from its own explorer/map and passes that, never a raw index).

**Consequence for `plan.md`'s `Mesh.__init__(shape)` design:** it needs to store more
than a bare `std::unique_ptr<SMESH_Mesh>` — it needs the id→`TopoDS_*` lookup tables too.
Cheap (same explorer pass `Shape` already ran), but it's a real change to the class's
internal state versus what B1's skeleton would naively assume.

### 1.4 `SMESH_ProxyMesh` (confirmed, no corrections)

`GetProxySubMesh(const TopoDS_Shape&)` takes a shape reference directly (consistent with
§1.3's fix), returns nullable `const SubMesh*` — B3 task 4 must null-check per wall face
before dereferencing, routing nulls into `failed_face_ids`. `NETGENPlugin_Mesher.cpp`
cross-reference (only available via `ref-freecad-smesh`'s sparse FreeCAD checkout, SMESH
~7.7.1-era, 2D-VL variant — `extern/smesh/` itself has no NETGENPlugin) confirms the
same "always resolve to `TopoDS_Face&` from your own explorer" idiom independently.

## 2. Repo-state gap — KERNEL is not vendored

`SMESHDS_Mesh.hxx` and `SMESH_ProxyMesh.hxx` both `#include <smIdType.hxx>`, a header
generated from SALOME KERNEL's `Basics/smIdType.hxx.in`, not part of `extern/smesh/` at
all. `plan.md`'s "Current Repo State" table and B1 task list only track `extern/smesh/`
— **KERNEL vendoring was missing from the plan entirely.**

Both `looooo_SMESH` and `ref-trelau-smesh` solve this via a KERNEL git submodule
(unfetched in this project's local clones) compiled down to a minimal slice:
`Basics/{Basics_Utils,Basics_DirUtils,BasicsGenericDestructor}.cxx`,
`SALOMELocalTrace/{BaseTraceCollector,FileTraceCollector,LocalTraceBufferPool,LocalTraceCollector}.cxx`,
`Utils/{duplicate,OpUtil,Utils_SALOME_Exception,Utils_ExceptHandlers}.cxx`, plus all
headers in those 3 dirs (including `Basics/smIdType.hxx.in`, `configure_file()`'d at
build time). Exact file list and CMake mechanism in
`docs/upstream_notes/SMESHDS_Mesh_notes.md` §"Gap discovered".

**Action required before B1's exit criterion can be met:** vendor this KERNEL slice as
a new `extern/kernel/` git subtree (tag matching SMESH `V9_9_0`, i.e. KERNEL `V9_9_0`),
recorded in `PROVENANCE.md`. Added as B1 open item #1 (see `PROVENANCE.md`
"Open items carried into B1").

## 3. Patch catalog — verified against upstream repositories

All 15 catalog entries from `plan.md` were checked for physical existence in upstream clones
(file existence + line count; patch *content*/applicability not
yet tested — that's B1). Full table with source repo, exact path, and commit is now in
`PROVENANCE.md` "Patch index" — not duplicated here to avoid drift between two copies.
One correction made during verification:

- The `SALOME_USE_64BIT_IDS` note in `plan.md`'s B0 "Important Note" cited
  `ref-freecad-smesh — SetupSalomeSMESH.cmake`. That file (and `FindSMESH.cmake`) **do
  not exist** in this project's local `ref-freecad-smesh` clone — it's a sparse checkout
  of only `src/3rdParty/salomesmesh/**`, and no `SALOME_USE_64BIT_IDS` string appears
  anywhere in it. The requirement is still correct and is now cited from a verifiable
  source instead (`extern/smesh/idl/SMESH_smIdType.idl.in` +
  `looooo/SMESH/cmake/Kernel/CMakeLists.txt:42`). See `PROVENANCE.md`
  "Citation correction".

All 15 patch source files/dirs were found present and non-empty except the above (which
was a citation for a *concept*, not a patch file, and is now re-sourced). No patch in the
catalog is missing from the reference clones.

Also confirmed independently: `ref-trelau-pysmesh/src/StdMeshers.cxx:283` already binds
`SetBndShapes` (not `SetFaces`) against SMESH 8.3.0.4 — i.e. this method name has been
stable since at least 8.3.0.4 through 9.9.0, so §1.1's correction is not a
version-drift artifact, `plan.md` simply had the wrong name from the start.
