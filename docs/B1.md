## 3. Tag-vs-pin skew (the recurring B1 friction) and how it's handled

Our `extern/smesh` and `extern/kernel` are the SALOME **V9_9_0 tags**; looooo's patches (and
conda's `0004`, cut on top of looooo) target looooo's V9_9_0 **branch pins**, which differ
slightly. Consequences and handling:

- **Patch application** (`prepare.py`) uses GNU `patch -N --fuzz=2`: `-N` auto-skips looooo
  source fixes already present in our newer tag (e.g. `SMESH_Controls.patch`,
  `SMESH_MesherHelper_msvc.patch` are logged as *SKIPPED (superseded)*); fuzz absorbs small
  offsets. conda's `0004` (99 hunks, OCCT-8.0) applies cleanly.
- **A few OCCT-8.0/Windows deltas looooo's pin already carried** are applied by `prepare.py`
  as documented idempotent fixups instead of patches (see `PROVENANCE.md` "B1 build reality"):
  `SALOME_LIGHT` (strip CORBA from KERNEL), `winsock2.h` for `gethostname`, `SMESH_TLink`
  default ctor + `operator()` hasher (OCCT-8.0 NCollection), `ElementsOnShape` out-of-line
  copy (incomplete-`Classifier` `std::vector`), `V3d_Coordinate`â†’`Standard_Real`, and the
  `TKSTL`â†’`TKDESTL` toolkit rename in CMake.

If B2/B3 hit further tag-vs-pin OCCT-8.0 breakage, the same pattern applies: add a documented
fixup in `prepare.py::_apply_tag_fixups`. (Re-vendoring at looooo's pins was considered and
rejected to keep clean official-tag provenance; the fixup surface has stayed small.)

## 5. How to build & test (local, `flux-pysmesh-build` env)

Requires the conda env (OCCT 8.0 / VTK 9.6.2 / Boost / cmake / ninja / pybind11 /
python-patch) **and** MSVC (`vcvars64`) + GNU `patch` on PATH.

```
python prepare.py                 # stage extern/ -> staged/ and apply patches (idempotent)
cmake -G Ninja -S . -B build -DCMAKE_BUILD_TYPE=Release \
      -DCMAKE_PREFIX_PATH=<env>/Library -DPython_EXECUTABLE=<env>/python.exe
cmake --build build -j
set PYSMESH_BUILD_DIR=build & pytest tests/test_hello_smeshds.py -v
```

`staged/` and `build/` are git-ignored. CI (B4) runs `python prepare.py` then
`python -m build`.
